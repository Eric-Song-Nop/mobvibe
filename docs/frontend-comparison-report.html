opencode 与 mobvibe Web 前端对比报告

结论：在当前代码状态下，opencode 的 Web 前端整体更成熟，优势主要体现在架构边界、共享 UI 系统、同步链路治理、国际化和文档站能力。mobvibe 并非低质量项目，它在 E2EE、Tauri/移动端入口、消息列表虚拟化和测试覆盖上有明确投入，但主应用前端仍有更强的耦合和重复实现。

评审日期：2026-05-11 方法：本地静态代码审查 + Web Interface Guidelines 规则抽样 未运行浏览器视觉回归，结论以代码证据为主

评审范围

opencode

  • packages/app：主要可交互 Web App。
  • packages/ui：共享 UI 组件、主题、Markdown、消息渲染、文件与图标系统。
  • packages/web：官网、文档站和分享页面。

mobvibe

  • ../mobvibe/apps/webui：主要 React Web UI，也是主比较对象。
  • ../mobvibe/apps/website：营销站、Pricing、Legal 页面。
  • ../mobvibe/docs：仓库内设计文档，未直接成为网站信息架构的一部分。

执行摘要

1. 架构边界更清晰 opencode 的入口层薄，Provider、路由、连接健康检查、会话级依赖分层明确；mobvibe 的 App.tsx 同时处理认证、路由、机器选择、会话选择、Socket、UI 派生状态和布局。
2. 设计系统更可复用 opencode 有独立 @opencode-ai/ui 包、主题 token、图标、弹窗、消息渲染和文件组件；mobvibe 的 webui 与 website 存在本地 UI 组件复制。
3. 同步与缓存更工程化 opencode 把 SSE 事件、coalescing、刷新队列、目录子 store、会话缓存和事件 reducer 分拆；mobvibe 的同步也强，但主要集中在大型 hook/store。
4. 国际化覆盖显著更广 opencode App 有 17 个 locale，文档站有 18 个 locale；mobvibe webui 和 website 当前各支持中英 2 种语言。
5. 官网/文档能力更完整 opencode 使用 Astro + Starlight + Cloudflare，内置多语言文档、编辑链接、配置 schema 生成和分享页；mobvibe website 更偏营销 demo。
6. mobvibe 的强项仍需保留 mobvibe 的 E2EE、Tauri 初始化、React Query、虚拟化列表和 E2E 测试是有效资产，后续改造不应推倒重来。

分项评审

1. 应用架构与职责边界

opencode 的 AppInterface 只负责装配服务器连接、健康检查、SDK、全局同步与路由。会话页再进入 SessionProviders，只加载 Terminal、File、Prompt、Comments 等会话级依赖。这个结构让全局依赖和会话依赖隔离，减少无关状态对整个应用的影响。

mobvibe 的 MainApp 聚合了认证态、URL 参数、机器发现、Socket、会话列表、UI store、命令面板、文件浏览器、热键、派生 loading/error 状态和主布局。虽然它已经拆出了一些 hooks，但入口组件仍然是业务编排中心，修改任何主流程都更容易碰到无关状态。

代码证据：packages/app/src/app.tsx 330 行；../mobvibe/apps/webui/src/App.tsx 953 行，并且在多个 route 分支中重复包裹 ThemeProvider。

工程判断：opencode 更符合 SOLID 的单一职责和依赖倒置。mobvibe 目前更像一个功能持续增长后的主控组件，短期能跑，长期更难稳定演进。

2. 状态同步、事件流与缓存治理

opencode 把同步链路拆成几个相对独立的层：GlobalSDKProvider 负责 SSE 事件流、心跳、重连、coalescing 和批量 flush；applyDirectoryEvent 是可测试的事件 reducer；createRefreshQueue 控制刷新节奏；createChildStoreManager 管理目录级 store、pin/unpin 和空闲淘汰；session-cache.ts 与 session-prefetch.ts 单独处理会话缓存和预取。

mobvibe 的 useSocket 也处理了很多复杂问题，包括 WAL backfill、revision mismatch、pending event、encrypted event buffer、跨 tab cursor 合并和 reconnect resubscribe。问题不在能力，而在这些能力集中在 1000 多行 hook 中，且大量依赖 ref、store action 和外部 socket。后续修同步 bug 时，影响面更难隔离。

代码证据：opencode 的 event-reducer.ts、queue.ts、child-store.ts、session-cache.ts 均有独立职责；mobvibe 的 useSocket.ts 同时处理事件应用、gap recovery、E2EE buffer、订阅管理和 reconnect。

工程判断：在高频 agent 消息流场景，opencode 的事件 coalescing、批量 flush、目录 store 生命周期和缓存淘汰更接近生产级 IDE 前端。mobvibe 的方案功能强，但需要继续拆分成纯 reducer、transport、backfill、subscription manager 四类模块。

3. UI 系统与主题能力

opencode 的 @opencode-ai/ui 是真正的共享 UI 层，包含 Button、IconButton、Dialog、Popover、List、Tabs、ScrollView、MessagePart、SessionReview、FileSearch、ThemeProvider、Markdown provider、文件图标与 provider 图标等。主题系统支持 JSON theme、light/dark variant、OKLCH 颜色推导、CSS token 输出、主题预览、命令面板切换和首屏 preload。

mobvibe 使用 shadcn/Radix 风格的本地组件。这个选择本身没有问题，但 webui 与 website 各自维护 UI wrapper，已经出现重复和细微漂移。例如两个 button.tsx 基本相同，但 webui 版本仍使用 transition-all，website 版本改成 transition-colors。这类差异会让修复分散，违反 DRY。

代码证据：packages/ui/src/theme/context.tsx 支持主题列表、预览、缓存和 storage 同步；packages/app/public/oc-theme-preload.js 有首屏主题注入；mobvibe 的 apps/webui/src/components/theme-provider.tsx 与 apps/website/src/components/theme-provider.tsx 是相近实现。

工程判断：opencode 的设计系统更适合长期维护、插件化功能和多入口复用。mobvibe 应优先抽出 @mobvibe/ui，再让 webui 和 website 共享同一套组件与 theme provider。

4. 面向开发者工作流的功能完整度

opencode 的交互面更像完整的开发工具：会话布局、会话标题/分享/删除、父子会话、终端面板、文件 tab、review tab、文件树、provider/model 管理、MCP 选择、LSP/MCP 状态、命令面板、prompt 附件、slash popover、上下文统计、主题和语言命令都存在明确模块。

mobvibe 也有机器侧边栏、会话列表、命令面板、文件浏览器、Git changes、commit history、E2EE 设置和移动/Tauri 相关能力，但当前 Web UI 更聚焦“远程多机器聊天控制台”。从纯开发者 IDE-like 使用面看，opencode 的 Web 前端能力更完整。

代码证据：opencode 的会话相关模块分布在 pages/session、components/session、components/prompt-input、components/file-tree.tsx、context/terminal.tsx；mobvibe 的对应核心集中在 App.tsx、ChatMessageList.tsx、ChatFooter.tsx、FileExplorerDialog.tsx。

5. 性能策略

opencode 的性能策略偏系统级：Solid 的细粒度响应式减少组件树重渲染压力；事件流按 16ms flush，并对 session.status、lsp.updated、message.part.updated 做 coalescing；时间线使用 staged mounting，避免一次性挂载大量历史 turn；目录 store 和 session cache 都有淘汰策略。

mobvibe 在局部性能上做得不错：ChatMessageList 用 @tanstack/react-virtual 虚拟化消息项，CommandPalette 在文件模式下也使用 virtualizer。它的问题更多是 App/store/hook 的组合复杂度会推高全局状态变化的认知成本，而不是缺少性能意识。

代码证据：opencode GlobalSDKProvider 有 FLUSH_FRAME_MS、STREAM_YIELD_MS、coalesced map；MessageTimeline 有 createTimelineStaging；mobvibe ChatMessageList 使用 useVirtualizer，overscan 为 8。

结论：opencode 在“全局事件吞吐和多目录资源治理”上更强；mobvibe 在“单个消息列表 DOM 数量控制”上也有优势，值得保留。

6. 可访问性与交互细节

两个项目都使用 Kobalte/Radix/Base UI 这类可访问性基础组件，方向是正确的。opencode 的 Dialog、Popover、Toast、TextField 等共享组件内置 i18n label，IconButton 使用 Kobalte Button，并在大部分调用点传入 aria-label。mobvibe 也有大量 aria-label，但存在更多本地 hardcode 和 UI wrapper 漂移。

Web Interface Guidelines 抽样发现，mobvibe webui 的基础 Button 包含 transition-all，并且存在 Upload image、Toggle Sidebar、Clear、Close 等硬编码英文 label；相对时间格式 just now、2h ago 也是硬编码。opencode 也不是完美状态，仍能看到 outline-none、部分 div/span onClick 和个别 IconButton 需要复查 label，但共享层让修复路径更集中。

代码证据：mobvibe apps/webui/src/components/ui/button.tsx 使用 transition-all；apps/webui/src/lib/ui-utils.ts 硬编码英文相对时间；opencode packages/ui/src/components/dialog.tsx 的关闭按钮使用 i18n 的 ui.common.close。

7. 国际化、内容和文档站

opencode App 语言层使用 lazy loader 合并 App 与 UI 包字典，并维护 INTL locale 映射。数字、时间和 token 统计多处走 locale-aware formatter。文档站基于 Starlight，配置了 18 个 locale、sidebar translations、edit link、last updated、social links 和构建后的 config schema 生成。

mobvibe 当前 webui 和 website 都只支持 zh/en。它的仓库内 docs 有不少中文设计文档，但 website 没有把这些内容组织成可发现的开发者文档站。对于面向开发者的产品，opencode 的 public web surface 明显更完整。

代码证据：opencode packages/app/src/context/language.tsx 支持 17 个 locale；packages/web/astro.config.mjs 配置 18 个 Starlight locale；mobvibe apps/webui/src/i18n/index.ts 的 supportedLanguages 为 zh 和 en。

8. 测试策略

opencode 的测试覆盖了不少基础设施边界：global sync、event reducer、session prefetch/cache、terminal writer、prompt submit/history/attachments、file tree、theme preload、i18n parity、layout scroll 等。mobvibe 的测试数量也不弱，尤其覆盖了 useSocket、useSessionMutations、ChatFooter、CommandPalette、FileExplorer、E2EE 和 session restore E2E。

结论不是“mobvibe 没测试”，而是 opencode 的测试更贴近它拆分后的内部契约；mobvibe 的同步核心仍然太集中，测试再多也会因为模块边界偏厚而难以定位回归。

代码证据：opencode packages/app 57 个测试文件、packages/ui 5 个；mobvibe apps/webui 49 个测试文件，包含 2 个 Playwright E2E spec。

mobvibe 的明确优势

本报告的主结论是“opencode Web 前端整体更成熟”，不是“mobvibe 应该重写”。以下能力是 mobvibe 应保留并继续强化的资产。

  • E2EE 与 Tauri/mobile 入口：mobvibe 对 encrypted payload、DEK bootstrap、Tauri storage、移动扫码和深链有明确设计，opencode Web App 不是同一个产品目标。
  • 消息列表虚拟化：ChatMessageList 的 @tanstack/react-virtual 是正确方向，长历史会话下 DOM 压力更可控。
  • React Query 使用：文件浏览器、分支选择、会话 mutation 等使用 Query/Mutation，有利于请求状态表达。
  • E2E 覆盖：session restore 和 E2EE 的 Playwright 测试说明 mobvibe 已经开始保护关键业务链路。

mobvibe 追赶建议

  1. 抽出共享 UI 包。 将 apps/webui/src/components/ui 与 apps/website/src/components/ui 收敛到 packages/ui 或 @mobvibe/ui，消除 button、theme provider、dialog 等重复。
  2. 拆薄 App.tsx。 保留路由入口，把主工作区拆成 AppProviders、AppRoutes、SessionWorkspace、SessionShell、SessionDialogs。不要在一个组件里同时派生所有 loading/error/action。
  3. 把 Socket ingest 做成纯模块。 将 useSocket 拆为 event reducer、cursor/backfill manager、encrypted buffer、subscription manager、notification adapter。React hook 只负责生命周期绑定。
  4. 补齐 i18n 和 formatter。 相对时间、aria label、toast、按钮文案统一走翻译表；日期数字走 Intl.DateTimeFormat 和 Intl.NumberFormat。
  5. 修复 Web Guidelines 抽样问题。 将 transition-all 改为明确属性；复查 outline-none 是否都有 focus-visible 替代；所有 icon-only button 使用本地化 aria-label。
  6. 把 docs 纳入 website 信息架构。 当前 ../mobvibe/docs 有有价值的工程文档，但 website 没有文档导航。可以采用 Starlight、Mintlify 或自建 docs route，先把 E2EE、Gateway、WebUI 性能和 ACP SDK 文档发布出来。

最终判断

opencode 的 Web 前端更好的核心原因不是“文件更多”，而是它把复杂度放在了更清晰的边界里：共享 UI 包承接视觉一致性，global SDK 和 reducer 承接同步复杂度，child store/cache 承接资源生命周期，language/theme/docs 承接公共体验。mobvibe 已经具备不少关键能力，但目前还缺少同等成熟的前端平台层。

如果 mobvibe 的目标是继续做多机器、E2EE、移动优先的 agent 控制台，它不需要复制 opencode 的所有功能；但需要学习 opencode 的边界拆分方式。优先级最高的是 DRY 的 UI 包、薄入口、纯同步 reducer、国际化 formatter 和文档站。
